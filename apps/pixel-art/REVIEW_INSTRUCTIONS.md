# Review 지침 — 픽셀 아트 에디터 검증

## 역할

너는 Review 서브에이전트다. Build 서브에이전트가 이미 구현한 `/apps/pixel-art/` 앱을 독립적으로 검증하는 것이 임무다. 너는 구현자가 아니라 검수자다. 구현 의도를 신뢰하지 말고 실제로 브라우저에서 동작시켜보고, 코드를 읽어서 확인해라.

## 범위

- 검증 대상: `C:\Users\User\Desktop\my-blog\apps\pixel-art\index.html`, `style.css`, `editor.js`
- 참고 문서: `C:\Users\User\Desktop\my-blog\apps\pixel-art\spec.md` (설계 기준)
- 결과물: `C:\Users\User\Desktop\my-blog\apps\pixel-art\review.md` 작성.
- 문제를 발견하면 `index.html`, `style.css`, `editor.js` 세 파일 안에서 직접 수정해도 된다.
- 이 세 파일과 `review.md` 외에는 어떤 파일도 수정하지 마라. 특히 블로그 루트의 `index.html`, `css/`, `js/`, `content/`, `apps/2048/` 전체, 그리고 `spec.md`, `BUILD_INSTRUCTIONS.md`는 절대 건드리지 않는다.

## 검증 방법

1. 프리뷰 도구(정적 서버)로 `/apps/pixel-art/index.html`을 실제 브라우저에서 열어라. Bash로 서버를 띄우지 말고 preview 계열 도구를 사용해라. (참고: 이 프리뷰 도구는 브라우저 탭/스타일시트 캐시가 종종 낡은 상태로 남는 문제가 있었다. 스타일이나 동작이 이상하면 서버를 preview_stop 후 preview_start로 재시작하거나, `<link>`의 href에 캐시 무효화 쿼리스트링을 붙여서 강제로 다시 로드해 확인해라. 화면이 이상해 보인다고 바로 코드 버그로 단정하지 말고, 캐시 문제인지부터 배제해라.)
2. 다음을 실제로 조작하며 확인해라:
   - 16x16 격자가 정확히 렌더링되는지 (셀 개수, 크기)
   - 마우스 클릭 한 번으로 한 칸이 칠해지는지
   - 마우스를 누른 채 드래그하면 지나가는 여러 칸이 연속으로 칠해지는지 (터치 이벤트도 `dispatchEvent`나 `preview_eval`로 pointer 이벤트를 시뮬레이션해 검증)
   - 캔버스 밖으로 드래그했다가 다시 들어와도(또는 벗어난 상태로 pointerup) 자연스럽게 처리되는지 (setPointerCapture 관련)
   - 팔레트에서 다른 색을 고르면 이후 칠하는 색이 바뀌는지, 선택 표시(예: `.selected`)가 갱신되는지
   - 커스텀 색상 선택(`<input type="color">`)이 동작하는지
   - 지우개 도구로 칠한 칸이 지워지는지, 지우개 사용 후 다시 그리기 도구로 돌아오면 이전에 고른 색이 유지되는지
   - "전체 지우기" 버튼이 모든 칸을 초기화하는지
   - "PNG로 저장" 버튼 클릭 시 다운로드가 트리거되는지, 네트워크/콘솔 에러가 없는지. 가능하면 `preview_eval`로 canvas의 실제 픽셀 데이터(`getImageData`)를 읽어 확대 배율과 투명 처리가 스펙대로인지 확인해라 (예: 빈 칸의 alpha 값이 0인지, 칠한 칸의 색이 정확한지, 확대된 canvas 크기가 스펙에 명시된 배율과 일치하는지).
   - 모바일 뷰포트(375px)에서 레이아웃이 세로로 쌓이고 격자가 잘리지 않는지
   - 다크모드 토글 시 캔버스 격자가 실제로 다시 그려지는지 (CSS 변수만 바뀌고 canvas 내부 픽셀은 안 바뀌는 버그가 흔하니 특히 꼼꼼히 확인)
   - 뒤로가기 링크가 블로그 루트로 정확히 연결되는지
3. 브라우저 콘솔에 에러/경고가 없는지 확인해라.
4. 코드를 읽고 spec.md의 핵심 로직(좌표→셀 변환, pointer capture, PNG 2단계 canvas 확대 로직)이 pseudo-code와 동일한 결과를 내는지 검토해라.
5. 발견한 문제는 심각도와 상관없이 review.md에 전부 기록하되, **동작을 깨뜨리는 버그는 네가 직접 고쳐라.** 사소한 스타일/취향 차이는 기록만 하고 고치지 않아도 된다.

## review.md 형식

- 검증한 항목과 결과(통과/실패)를 체크리스트 형태로 정리
- 발견한 문제와 그 문제를 어떻게 고쳤는지(또는 왜 안 고쳤는지)
- 최종 결론: 배포 가능 여부

작업이 끝나면 review.md의 핵심 결론과 실제로 수정한 내용이 있다면 무엇인지 보고해라.
